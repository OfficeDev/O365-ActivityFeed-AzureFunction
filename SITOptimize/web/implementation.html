How DLP tuning works in Purview

The agent's plan follows the two paths below. Keep this open as a reference while you apply changes. Portal labels can differ slightly between compliance.microsoft.com and the Microsoft Purview portal.

A SIT is an entity definition

Every SIT is built from a primary element (a regex, a keyword list, a dictionary, or a built-in function such as Func_credit_card), optional supporting elements (keywords/regex that must appear within a character proximity window), and confidence levels. Confidence maps to an accuracy number: Low = 65 or below, Medium = 75, High = 85. There are three places you can tune, plus the DLP rule itself.

The pattern editor gives you precise knobs — don’t under-use them:

  • Element types (primary & supporting): regular expression, keyword list, keyword dictionary, or function.
  • Element groups: Any of these (OR), All of these (AND), and Not any of these — a real negative / exclusion group (“match unless one of these noise terms is also present”).
  • Character proximity: detect primary + supporting within N characters (tighten, e.g. 150–300), or Anywhere in the document.
  • Additional checks: Exclude specific values, Starts / doesn’t start with, Ends / doesn’t end with, Exclude duplicate characters, Include / exclude prefixes, Include / exclude suffixes — surgical, low-risk FP cutters.
  • Confidence level per pattern (Low / Medium / High).

First: is the SIT built-in or custom?

Well-known Microsoft names (“Credit Card Number”, “U.S. Social Security Number (SSN)”, “Azure Storage Account Key”, bundles like “All Full Names”) are built-in; anything created in your tenant is custom. Under Data classification → Classifiers → Sensitive info types, custom types have an Edit button — built-in types open read-only but offer a Copy action.

Tier 1 — Editing a built-in in place: not possible

You can’t change a Microsoft-provided SIT as shipped. Don’t try to edit the original — use Tier 2 (copy & tune) or the DLP rule controls instead.

Tier 2 — Copy the built-in and tune the copy (preferred for most built-ins)

Open the built-in SIT → Copy → you now have an editable custom copy. For detections like Credit Card Number this is the recommended fix. On the copy you can:

  1. Adjust supporting elements — add corroborating keywords/ dictionaries (e.g. credit, card, cvv, exp) or remove the ones driving noise; combine them as Any / All / Not any of these.
  2. Add a “Not any of these” group to exclude context-specific noise (test banners, sample-data markers) without weakening the primary detection.
  3. Tighten character proximity so the primary match only counts when that context is nearby (e.g. 150–300 characters).
  4. Add additional checks — Exclude specific values (known test/sample literals), Exclude duplicate characters (kill 0000…/1111…), require/forbid prefixes & suffixes, starts/ends-with constraints.
  5. Raise confidence / accuracy (Low ≤65 / Medium 75 / High 85) and point the DLP rule at High.
  6. Function-based primaries (Func_*) can’t be rewritten, but everything around them is tunable. If the copy’s primary is a plain regex you can also tighten that regex.

Then repoint the DLP rule from the built-in to your tuned copy. The original built-in stays untouched, so rollback is instant. Path: Sensitive info types → [built-in] → Copy → edit → Test → Save / Publish.

Tier 3 — Custom SITs (fully editable)

Edit the definition directly. Cut false positives by requiring more corroboration and by using the editor’s built-in exclusions — Not any of these groups and the Additional checks (Exclude specific values, doesn’t-start/end-with, exclude prefixes/suffixes).

  1. Choose the primary element (regex / keyword list / dictionary / function). For a regex, tighten it — anchor it, restrict character classes, add word boundaries, require structure. Keep the regex forward-matching only: don’t use lookbehind ((?<=…) / (?<!…)) or lookahead in the primary pattern — Purview’s engine doesn’t reliably support them and they can silently break the whole match, dropping real true positives. Do all exclusion logic outside the regex (next point).
  2. Require supporting elements + groups within a character proximity window; use All of these for more corroboration and Not any of these to exclude noise terms near the match.
  3. Add additional checks — Exclude specific values, Exclude duplicate characters, starts/ends-with, include/exclude prefixes & suffixes — for surgical FP removal.
  4. Use confidence levels — corroborated matches High, weak matches lower.
  5. Curate keywords — add corroborating, remove over-broad, use a dictionary when large.

Path: Sensitive info types → [custom SIT] → Edit → edit pattern → Test → Save / Publish.

Always available — DLP rule controls (any SIT type, no SIT edit)

These work whether the SIT is built-in or custom and are the fastest, most reversible wins:

  1. Raise the minimum instance count (e.g. require 5–10 matches instead of 1).
  2. Require higher confidence on the SIT condition (Low → Medium → High).
  3. Add predicate exceptions directly in the rule (Add exception / NOT conditions) — e.g. except when content contains noise keywords (test, sample, example, dummy).

Path: Data Loss Prevention → Policies → [policy] → Edit policy → [rule] → Edit → Conditions / Exceptions.

Exceptions are workload-dependent

The predicates you can use in a rule exception depend on the workload the rule covers. Match the exception to where the noise is coming from (the analysis reports a source_workload for each pattern) — an email-only predicate won’t exist on a SharePoint or Endpoint rule:

  • Exchange / email: except by sender / recipient (domain, group, member), sender IP range, subject/body/header words or patterns, attachment type/size.
  • SharePoint / OneDrive: except by document property / managed metadata, site / library / path, file name or extension, sensitivity label.
  • Teams (chat & channel): except by sender / recipient (internal vs external), or content contains words / patterns.
  • Endpoint / Devices: except by file path / extension, app or browser, network / domain. (Most email predicates like sender/recipient don’t apply here.)

Keep exception terms specific — or you suppress real findings

A “content contains words / phrases” exception is a substring match over the whole item. A short or fragmented term silently excludes huge volumes of legitimate content — suppressing real true positives, which is worse than a false positive. Guard rails:

  • Never use fragments or structural snippets as exception terms — e.g. &p=, ?e=, id=, LinkId=, destination=, or single words like link, token, key. They appear everywhere and gut detection.
  • Don’t split a noise string into pieces. Only the full, distinctive string is safe — matching the parts matches almost everything.
  • Prefer a structural predicate over a content keyword whenever one exists for the workload (sender / URL domain, site, path, file extension). It’s precise and doesn’t scan the whole body.
  • If you must use a keyword, make it long, unique and anchored (a full FQDN like safelinks.protection.outlook.com or a complete unique phrase) and confirm it essentially only ever appears in the noise source.

Always: test, roll out safely, and measure

  • Test the SIT with the built-in Test tool (paste / upload sample content) before publishing.
  • Run the policy in simulation / test mode (alerts without enforcement) first, then enforce once the volume looks right.
  • Allow propagation. SIT and rule changes can take up to ~1 hour (longer for data-at-rest, which needs re-crawl) to take effect.
  • Keep a rollback. Built-in originals are never edited; tune copies or use test mode so you can revert instantly.
  • Quantify the win. Capture a run before and after, then use the Delta comparison page to measure the noise reduction.
