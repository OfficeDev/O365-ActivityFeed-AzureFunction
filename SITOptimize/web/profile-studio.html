SIT Tuning Studio

Profile Studio

Two tracks in one place. Run & compare: run any saved profile to produce a run snapshot and measure noise reduction on the delta page. Create a new profile: describe any sensitive information type and let the agent draft a noise-reduction tuning profile — the prompts exist to find why the SIT is over-firing and how to suppress the false positives while protecting genuine true positives. Edit the language, test it, then export a ready-to-run .psd1 for credpattern.ps1.

Setup

Server connection

This page is served by studio-server.ps1, a local PowerShell web server. It runs the agent, test runs, file saves and the full pipeline on your machine using $env:AZURE_OPENAI_API_KEY — the key never reaches the browser.

Not connected to the studio server. Start it from the repo root and reload this page: 
$env:AZURE_OPENAI_API_KEY = '<your-key>'
pwsh .\studio-server.ps1
Then browse to http://localhost:8787/.

Leave the endpoint blank to use the server's default. Point it at your own Azure OpenAI resource to use a different endpoint — the matching AZURE_OPENAI_API_KEY must be set on the server that resource expects.

Run & compare

Run an existing profile

Already have a profile under profiles/? Pick it and run the pipeline directly to produce a fresh run snapshot — then compare a baseline run against a post-tuning run on the delta page. No need to regenerate the language. (Use the Create a new profile track below only when building or editing the tuning language itself.)

Create a new profile · Step 1

Describe the sensitive information type

A profile teaches the AI how to read this SIT's DLP matches — so it can tell real sensitive findings from the noise (false positives) and recommend tuning. Just say what the SIT is; the agent infers the rest: the policy filter, the finding noun, the entity types/formats, the likely false-positive sources and the validity signals. Create a new profile · Step 2

Review & edit the generated profile

The JSON container keys in the extraction prompt (noise_patterns, validated_credentials, credential_pairs…) are kept identical across every profile so the delta tool and report consolidation keep working. Change the wording, examples and risk guidance freely — keep those keys.
